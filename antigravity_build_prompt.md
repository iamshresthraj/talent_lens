## 1. Hard constraints (treat as non-negotiable)

- Ranking script (`rank.py`) must run in ≤5 min, ≤16GB RAM, CPU-only, no network,
  no GPU, ≤5GB intermediate disk. Precomputation steps (embeddings, indexes) are
  exempt from the 5-minute limit but must be scripted and reproducible, and must be
  clearly separated from `rank.py` in the repo.
- No hosted LLM API calls anywhere in the ranking path (no OpenAI/Anthropic/Cohere/
  Gemini calls). A local, CPU-quantized model (e.g. via `llama-cpp-python`) is
  allowed but must be optional and guarded — if it's not installed or would risk
  the time budget, fall back automatically to the deterministic template writer.
  Set `HF_HUB_OFFLINE=1` / equivalent in `rank.py` so embedding model loading can
  never silently trigger a network call.
- Output CSV columns, in order: `candidate_id,rank,score,reasoning`. Exactly 100
  data rows. Ranks 1–100, each used once. Scores non-increasing with rank, ties
  broken by `candidate_id` ascending. Validate with the provided
  `validate_submission.py` before declaring done.
- Filename for final submission: `submission.csv` (rename instructions for the
  actual participant ID go in the README).
- No login, no authentication, no user accounts anywhere in this project,
  including the sandbox app. This is a reviewer-facing demo tool, not a product
  with users — keep it open and frictionless to run.

## 5. Scoring design — implement exactly this in `src/scoring.py`, weights in `config/weights.yaml`

All sub-scores normalized to [0,1] before combining.

```
linear_score =
    0.28 * semantic_fit          # cosine(jd_embedding, candidate_embedding), min-max normalized across pool
  + 0.20 * skill_depth_fit        # must-have terms found, weighted by proficiency (beginner..expert -> 0.25/0.5/0.75/1.0)
                                  #   * log1p(duration_months) * presence_of_endorsements_bonus, then weighted-summed
                                  #   per must_have_skills weight and normalized
  + 0.12 * lexical_fit            # TF-IDF/BM25 score of full candidate doc against JD must+nice-have term list, normalized
  + 0.30 * structural_fit         # see rules.py below — production evidence, experience-band fit, soft-disqualifier
                                  #   penalties already folded in here as subtractive terms before normalization
  + 0.10 * logistics_fit          # location/relocation + notice period + (mild) salary-band sanity

final_score = linear_score * behavioral_multiplier

# behavioral_multiplier in roughly [0.65, 1.25], built from:
#   recency_factor: last_active_date decay (full credit if active <14d, decaying to ~0.7 by 180d+)
#   open_to_work_flag: +0.05 if true
#   recruiter_response_rate, interview_completion_rate: scaled contribution, weight ~0.1 each
#   offer_acceptance_rate: treat -1 (no history) as neutral 0.5, not as 0
#   profile_completeness_score: small contribution
#   verified_email/phone/linkedin_connected: small completeness bonus (+0.02 each, capped)
#   github_activity_score: small bonus if >0, treat -1 (no GitHub) as neutral, not penalized

# Hard disqualifier triggered  -> exclude candidate from ranking entirely (do not even compute final_score)
# Soft disqualifier triggered  -> final_score *= 0.15 to 0.30 (pick the multiplier per rule, document in rules.py)
# Honeypot triggered           -> exclude candidate from ranking entirely (separate from disqualifiers, see section 6)
```

Note the deliberate asymmetry: the JD itself uses "we will not move forward" for
exactly one disqualifier (pure research, zero production) and "we will probably
not move forward" for the others — mirror that distinction in code (hard exclude
vs heavy penalty) rather than treating all negatives the same. This is a detail
worth being able to explain in the interview stage.

## 6. Honeypot detection — implement in `src/rules.py` as `detect_honeypot(candidate) -> bool`

Flag a candidate as a honeypot if **2 or more** of these fire (tune threshold after
checking the actual flag rate lands well under 10% of the pool, ideally well under
1% since these should be rare):

- Any skill with `proficiency == "expert"` and `duration_months < 6` — and 2+ such
  skills is an even stronger signal on its own.
- `abs(sum(career_history.duration_months)/12 - years_of_experience) > years_of_experience * 0.3`
  (and > 1 year absolute, to avoid penalizing small rounding).
- More than one `career_history` entry with `is_current == true`, or more than one
  entry with `end_date == null`.
- Any `career_history` entry where `end_date < start_date`.
- `education[].end_year` later than the start_year of the candidate's *first*
  career_history entry by an implausible margin (degree finishing after they'd
  already supposedly worked for years), with no other degree explaining it.
- `expected_salary_range_inr_lpa.min > expected_salary_range_inr_lpa.max`.
- `years_of_experience` very high (>10) while `education[].end_year` is very recent
  (<3 years ago) and there's no earlier degree.

Write 6-8 unit tests in `tests/test_rules.py` with synthetic candidates covering
each rule individually (one rule firing should NOT flag a candidate; two rules
firing should).

## 7. Reasoning generation — `src/reasoning.py`, single interface, two backends

```python
def generate_reasoning(candidate: dict, facts: dict, mode: str = "template") -> str:
    ...
```

- `facts` is a small dict already extracted by the scoring step: years_of_experience,
  current_title, current_company, top 2-3 matched must-have skills with their
  proficiency, one concern if applicable (notice_period_days, location mismatch,
  a soft-disqualifier that fired but didn't exclude them, low recruiter_response_rate),
  and last_active recency in human terms ("active this week" / "inactive 4 months").
- `mode="template"` (the default used by `rank.py` for the actual submission):
  assemble 1-2 sentences from `facts` only, using a small bank of interchangeable
  sentence templates/connectives selected deterministically by
  `hash(candidate_id) % N` so wording varies across the 100 rows without ever
  inventing a fact. Never reference a skill/company/number not present in `facts`.
- `mode="llm"` (optional, used only in `sandbox/app.py` where N≤100 and latency is
  cheap): if `llama-cpp-python` and a small local GGUF model
  (e.g. Qwen2.5-0.5B-Instruct or similar ~0.5-1.5B quantized model, downloaded once
  in `00_setup_models.py`) are available, prompt it with a strict system message:
  "Using ONLY the facts listed below, write exactly 1-2 sentences explaining
  whether this candidate fits the role. Do not mention any skill, company, or
  number that is not explicitly listed. If there is a concern listed, mention it
  honestly." Wrap in try/except — any failure (missing model, import error,
  timeout) falls back to `mode="template"` silently.
- `rank.py` must NOT default to `mode="llm"` for the graded run — explain this
  choice in the README (time-budget risk on an unknown grading machine running
  100 sequential generations vs. a guaranteed sub-second template path that
  already passes all six Stage-4 reasoning checks).
